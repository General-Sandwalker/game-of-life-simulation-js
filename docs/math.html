The Prisoner's Dilemma

The Prisoner's Dilemma (PD) is the foundation of EvoBlob. It models a situation where two rational agents each choose to either Cooperate (C) or Defect (D) simultaneously, without knowing the other's choice.

The paradox: mutual defection is the only Nash Equilibrium (neither player can improve by changing alone), yet mutual cooperation yields a higher total payoff for both players. This tension between individual and collective rationality produces rich evolutionary dynamics.

Nash Equilibrium: A state where no player can improve their payoff by unilaterally changing their strategy, given that all other players' strategies remain fixed.

Payoff Matrix

Four constants define a Prisoner's Dilemma. The constraint T > R > P > S must hold for the game to be a true PD.

Opponent: C
Opponent: D
Me: C
R, R
Reward
S, T
Sucker / Tempt.
Me: D
T, S
Tempt. / Sucker
P, P
Punishment

The cell format is (my payoff, opponent payoff). In code:

function calcPayoffs(moveA, moveB, { T, R, P, S }) {
  if (moveA === 'C' && moveB === 'C') return [R, R];
  if (moveA === 'C' && moveB === 'D') return [S, T];
  if (moveA === 'D' && moveB === 'C') return [T, S];
  return [P, P]; // both defect
}

Iterated PD

EvoBlob uses the Iterated Prisoner's Dilemma (IPD): two blobs play multiple rounds (roundsPerMatch) against each other. This is crucial because it allows memory-based strategies (like Tit-for-Tat) to outperform pure defectors over time.

ScoreA += Σr=1..R payoff(moveAr, moveBr) × scale

Where scale normalises payoffs (see Score Scaling).

Strategy Algorithms

All strategies are pure functions with signature:

decide(memory, opponentId, params) → 'C' | 'D'

Where memory is an ordered array of past rounds: [{myMove, opponentMove}, ...].

Always Cooperate / Always Defect

Trivial baseline strategies. No memory required.

decide(·) = C    or    decide(·) = D

Tit-for-Tat (TFT)

Introduced by Anatol Rapoport, TFT won Axelrod's famous computer tournaments in the 1980s. It is nice (never defects first), retaliatory (punishes defection immediately), forgiving (returns to cooperation once the opponent does), and clear (simple to understand).

decide(memory) = { C   if |memory| = 0,   memory[last].opponentMove   otherwise }

Tit-for-Two-Tats (TF2T)

Defects only after the opponent defects twice in a row. More forgiving than TFT.

decide(memory) = D   iff   memory[-2].opponentMove = D  &&  memory[-1].opponentMove = D

Grudger (Grim Trigger)

Once the opponent defects, Grudger defects forever. It is the harshest form of punishment. In evolutionary tournaments it often outperforms TFT in very stable populations but collapses when playing against other Grudgers in a noisy environment.

decide(memory) = { C   if ∀r: memory[r].opponentMove = C,   D   otherwise }

Pavlov (Win-Stay Lose-Shift)

Pavlov repeats its previous move if the outcome was "good" (payoff > P), and switches otherwise. This strategy can exploit accidental cooperators and self-correct out of mutual defection.

last_payoff = payoff(last.myMove, last.opponentMove)
decide = last.myMove   if last_payoff > P   else   flip(last.myMove)

Random

decide(·) = C with probability 0.5, D with probability 0.5

Generous Tit-for-Tat (GTFT)

Cooperates in response to defection with a forgiveness probability p (default 10%). This prevents getting locked into mutual defection spirals with noisy partners.

decide(memory) = { C   if last_opponent_move = C,
               C with prob. p  |  D with prob. (1−p)   if last_opponent_move = D }

Detective

Runs a 4-round probe sequence (C, D, C, C). If the opponent never retaliates during the probe, Detective switches to Always Defect (the opponent is too passive). Otherwise it falls back to TFT. 

// Round 0..3: always play PROBE[round] = [C, D, C, C]
// Round 4+:
//   if opponent never defected during probe → Always Defect
//   else → Tit-for-Tat

Soft Majority

Cooperates if the opponent cooperated in at least half of all remembered rounds. More tolerant than Grudger but reactive to sustained defection patterns.

decide = C   iff   |{r : opponentMover = C}| / |memory| ≥ 0.5

Memory Model

Each blob maintains a per-opponent memory — a separate ordered history for every blob it has encountered. This is stored as a JavaScript Map<blobId, Array<...>>.

When two blobs meet, each records the round as: 

blob.memory.get(opponentId).push({ myMove: 'C', opponentMove: 'D' });

The memory is trimmed to the last memoryLen entries using a sliding window: 

if (history.length > memoryLen) {
  history.splice(0, history.length - memoryLen);
}

When a blob dies its memory is garbage-collected. When a new blob is born it starts with an empty memory — it has no inherited knowledge of other blobs.

Memory length is a critical parameter. With memoryLen = 1, TFT and Grudger behave identically (both react only to the last move). With longer memory, Grudger remains permanently unforgiving while TFT can "forget" betrayals.

Evolutionary Dynamics

Selection (Natural)

EvoBlob uses implicit selection — there is no explicit fitness-based selection algorithm. Instead, blobs with higher scores survive longer and reproduce more, while low-scoring blobs die. This mimics natural selection:

  • Blobs with score < deathThreshold → die (selection pressure)
  • Blobs with score ≥ reproThreshold → reproduce (differential reproduction)

Because payoffs come from the game interactions, a strategy's fitness depends on the current composition of the population. This creates frequency-dependent selection: a strategy can be advantageous when rare but disadvantageous when common.

Reproduction

Reproduction is asexual — a single parent creates one offspring. At each generation end, any blob above the threshold reproduces:

parent.score ← parent.score / 2
child.score ← parent.score   (same value post-split)

The child inherits the parent's strategy and spawns at a slightly randomised position near the parent. This energy split prevents unbounded score growth.

Population cap enforcement: Reproduction is attempted in a snapshot of the pre-reproduction population, and halted if the cap is reached. This prevents exponential growth.

Mutation

Each newborn has a mutationRate probability of adopting a uniformly random different strategy: 

const mutated = Math.random() < params.mutationRate;
const childStrategyId = mutated
  ? randomStrategyExcept(parent.strategyId)
  : parent.strategyId;

Mutation is the source of genetic variation. Without it, lost strategies can never return. With too high a mutation rate, evolutionary pressure is swamped by noise and no strategy reaches dominance.

P(mutation) = μ    where μ ∈ [0, 1]

Optimal mutation rates in evolutionary game theory are problem-dependent. In most PD settings, a small μ (1–10%) allows exploration without destroying selection pressure.

Score Scaling

Raw payoff values (T, R, P, S) define the relative advantage of each move combination. Without scaling, high match counts would cause runaway score inflation and all blobs would reproduce every generation de-coupling scores from selection.

EvoBlob divides each payoff by:

scale = 1 / (matchesPerGen × roundsPerMatch)

This means a generation of maximum cooperation (all R payoffs) contributes exactly R points total to a blob's score — regardless of how many matches or rounds are configured. A blob starting at score initScore = 5 needs to accumulate 5 more net points to reach the reproduction threshold of 10.

Simulation Loop

Each generation runs the following pipeline:

for each generation:
  1. MATCHES
     shuffle(blobs)
     for m in 1..matchesPerGen:
       pair blobs sequentially (odd blob sits out)
       for each pair (A, B):
         for r in 1..roundsPerMatch:
           moveA = A.decide(B.id, memory, params)
           moveB = B.decide(A.id, memory, params)
           [pa, pb] = calcPayoffs(moveA, moveB, payoff)
           A.score += pa × scale ; B.score += pb × scale
           A.recordRound(B.id, moveA, moveB)
           B.recordRound(A.id, moveB, moveA)

  2. REPRODUCTION
     for each blob (snapshot):
       if blob.score >= reproThreshold and pop < maxPop:
         child = blob.reproduce(mutationRate)
         blobs.push(child)

  3. AGE & DEATHS
     for each blob:
       blob.age++
       if blob.score < deathThreshold:
         remove from population

  4. SNAPSHOT
     record counts, avgScores per strategy
     push to history array

  5. EMIT
     fire 'generation' event → update UI, canvas, charts

Pairing Algorithm

Blobs are paired using a Fisher-Yates shuffle followed by sequential pairing. This gives O(n) pairing complexity with uniform random pair selection: 

// Fisher-Yates in-place shuffle
function shuffle(arr) {
  for (let i = arr.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [arr[i], arr[j]] = [arr[j], arr[i]];
  }
  return arr;
}

Rendering Loop

The canvas renderer runs its own requestAnimationFrame loop, independent of the simulation loop. This decouples visual smoothness from simulation speed — the canvas always renders at 60fps; the simulation ticks multiple generations per frame when speed is high.

Rendering

Blob Visualisation

Each blob is rendered as a layered canvas arc:

  • Glow layer — radial gradient at 1.5× radius, fades out for newborns (intensity decays 2%/frame).
  • Body gradient — radial gradient from a lighter highlight offset top-left to a darker base.
  • Score label — shown if radius > 9px.

Physics (Soft Random Walk)

Blobs move using a simple velocity-damping model with boundary reflection. Positions are stored in normalised coordinates [0, 1] and mapped to canvas pixels at render time, so resizing the canvas does not disrupt positions: 

blob.vx += (Math.random() - 0.5) * SPEED_BASE;
blob.vy += (Math.random() - 0.5) * SPEED_BASE;
blob.vx *= DAMPING;   // DAMPING = 0.95
blob.vy *= DAMPING;
blob.x  += blob.vx;
blob.y  += blob.vy;
// Boundary reflection
if (blob.x < pad) { blob.x = pad; blob.vx =  |blob.vx|; }

Blob Radius

r = rmin + min(score / 10, 1) × (rmax − rmin)

where rmin = 4px and rmax = 18px.

Idle Pulse Animation

Each blob oscillates its drawn radius with a per-blob phase offset:

rdrawn = r × (1 + 0.04 × sin(φ + t × 1.5))

where φ is randomised at birth and t is elapsed time in seconds.

Computational Complexity

OperationComplexityNotes
Shuffle (pairing)O(n)Fisher-Yates; n = population
Matches per generationO(n × M × R)M = matchesPerGen, R = roundsPerMatch
Reproduction scanO(n)Linear pass over snapshot
Death scanO(n)Linear filter
Strategy stats snapshotO(n)Counted in one pass

Chart updateO(H)H = history length (capped at 1000)
Memory per blobO(k × L)k = unique opponents met, L = memoryLen

Dominant cost: the matching phase at O(n × M × R). With defaults (n=60, M=10, R=5), this is 3000 strategy invocations per generation — trivial for modern CPUs. At n=300, M=50, R=20 it becomes 300,000 invocations/gen; still fast in JavaScript but may limit Speed at very high values.
